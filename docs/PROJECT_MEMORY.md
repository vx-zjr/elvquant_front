# elvquant_front Project Memory

This document is the durable working memory for `elvquant_front`. Read it before each task and
update it whenever the product direction, architecture, workflow, or validation contract changes.

## Product Direction

`elvquant_front` is the local research cockpit for `elvquant_core`. It should feel like an
institutional quant research workstation: dense, calm, fast to scan, precise with status, and built
for repeated research/debugging workflows rather than marketing presentation.

Design direction after 2026-06-11 review: UI/UX quality should lead backend alignment for the next
phase. The cockpit should feel closer to a top-tier quant research and trading workstation even when
some panels are UI-derived or deterministic placeholders from the current API state.

The current target is a React + TypeScript cockpit backed by a thin Python API. The UI should make
these workflows first-class:

- inspect available public `elvquant_core` workflows
- run one workflow at a time through public core runner functions
- view parsed metrics, equity rows, report text, and raw output
- inspect Stooq readiness without leaking tracebacks or secrets
- browse local markdown reports generated by core
- keep operator-visible boundaries clear: UI selects, triggers, and renders; core owns business logic

## Architecture Decisions

- Frontend: Vite, React, TypeScript, dense dashboard layout, custom CSS, lucide-react icons.
- Backend: FastAPI served by `elvquant_front.api:app`.
- Core bridge: Python module isolates all public `qts.*` imports and report parsing.
- Static serving: FastAPI serves `web/dist` when a production build exists; Vite proxies `/api` in
  development.
- Secrets: no API keys or credentials in the frontend, API responses, committed config, or docs.

## Boundary Rules

- Allowed core imports are public runner/config APIs only.
- The frontend must not instantiate or reimplement strategy, risk, broker, accounting, order, signal,
  or portfolio objects from core.
- New business behavior must be implemented and exposed from `elvquant_core` first, then consumed by
  this frontend.
- Missing core/data/config states should be rendered as operator-friendly status, not raw tracebacks.

## Development Workflow

Every material task should follow this loop:

1. Read this file, `docs/ITERATION_LOG.md`, and the active plan.
2. Add or update tests before changing contract-sensitive behavior.
3. Implement the smallest coherent change.
4. Run Python tests/lint and frontend type/build checks.
5. Update docs with decisions, commands, validation output, and remaining gaps.
6. Commit intentionally and push the working branch or requested target branch.

## Validation Commands

Use these as the default gate when the relevant toolchain is installed:

```powershell
python -m pytest
python -m ruff check
python -m pip install -e ".[dev]"
npm ci
npm run build
```

On this Codex desktop host, Python and Node may need explicit bundled paths when system commands are
not on `PATH`. For npm on Windows, `.nodeenv` works when `.nodeenv\Scripts` is prepended to `PATH`.
